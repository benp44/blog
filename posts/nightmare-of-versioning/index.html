On the problems of software versioning

If you'll indulge me, a metaphor…

Software versioning has always seemed to me to be one of the thorniest issues of software engineering. I'll try to explain why using a metaphor.

Think of a large software project you're familiar with. Now think about some of the details of that software; e.g. the different classes, modules, etc; some of the simpler and more complicated parts of the code; some of the known issues and bugs; any concurrency that might be involved; and of course some of those dark places in the code that are best left untouched. With this in mind, let's imagine the project as a landscape; something you can look across and recognise the features. The landscape has different ‘terrains’ - some safe and easy to navigate, some less so.

Once you've been working on a large project for a while, your brain builds up an abstraction or mental model of the code. You don't know every line of code off the top of your head, but the abstraction allows you to generalise and recollect the different parts of the code that I suggested above. In our landscape metaphor, we could relate this abstraction to a map. Having this map in your head allows you to navigate the landscape confidently.

Of course (being an engineer) you not only want to navigate the landscape, but to modify it too. Fortunately when you do this, you can also update the map in order to keep it up to date. Furthermore, if you are working in collaboration with others on this landscape, and you are able to communicate well, when others discover new features or make changes to the landscape, everyone can update their maps. Maybe not precisely, but at least with some marker or note.

“But what does all this have to do with software versioning?”

Creating a version of software is like taking a snapshot of the source code at a point in time, and thus we can (just about) extend this to our metaphor - were we to freeze the state of our landscape at a point in time.

But can we do the same with our map? As it turns out, not so much. We now have to have multiple maps to keep track of each version of the landscape if we want to navigate them as before. And we often do want to do this - not necessarily out of choice, but because we potentially need to maintain each version. We've added an extra dimension to our landscape - how it changes over time - and this is exponentially more information for us to map and navigate.

The metaphor is straining a bit now, but I hope you get the point. Unlike software stored in a version control system, our brains don't have a perfect recollection of every change and version of the software that exists. But then the version control system does not provide the abstraction we built in our heads. At least not an easily re-collectable one. We have fixed limits on our cognitive load and what we can usefully remember, and these limits don't scale when it comes to exponential growth.

Whenever I need to go back and revisit some version of software for a patch or otherwise, this problem manifests itself as a familiar cortisol release. I don't know which aspects of my mental model I have for the project are accurate and which are going to betray me because the code was different when it was versioned.

So… what's the solution?

There is no easy solution to this problem, versioning is an essential part of the software development process. But from my experience there are some practices that can help.

Remember: Agile does not mean no documentation at all

We can aid our memories by writing documentation. In fact this is the default solution to this problem - as analogous as we can get to keeping and maintaining multiple “maps”. But then in our agile software development world, producing large quantities of documentation is not desirable. But producing some documentation certainly is. The mistake often made by engineer-led companies and teams is to use agile as an excuse to avoid that most boring of tasks, documentation.

Version documentation alongside code

What documentation you do have should be versioned just like code. Now when you need to patch some old version of your project, you know where to look for the documentation relevant to that version of the code.

Commit messages are a special case of this. If you haven't learned to write good commit messages yet, you will do soon. Squash and re-write if necessary.

Try to avoid unnecessary versioning

While we can't avoid versioning, we can try to reduce the proliferation of versions. Remember that when we talk about versions, this includes branches and tags. Try to be mindful about how you use your source control system. I would always advocate trying to stay close to the master branch wherever possible. It's all too easy for branches to drift away from master resulting in conflicts and confusion when the time does come to merge. “Commit often” is a well-known idiom, but it also pays to merge often too.

I've heard experienced developers boast of the small set of git commands they use - essentially pull, commit and push. If you are using advanced versions of your source control system often, it may be a warning sign that things are getting too complicated.

Why have multiple repositories when one will do?

When using a tool like Github, it's so easy and seemingly correct to put each project into its own repository (in my current job I work with something like 50 of them). But if projects have hard or soft dependencies between them, it can actually pay to keep them all together in a single repo, just using subdirectories to separate them. This way you pick up other's changes much sooner and know much sooner when you break something. You can also see an interwoven timeline of changes across projects by listing commits in the root - and still see a projects changes independently by listing in each sub-directory.

This isn't always going to work - for sure there are lots of good reasons for modularity - but it's worth consideration.
